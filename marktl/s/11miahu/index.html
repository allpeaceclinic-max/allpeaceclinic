Section 01

한 장면으로 보는 사례

추상적인 이론 대신, 지금 실제로 돌아가는 것 세 가지부터.

이사회 준비
단톡방 메시지·공유 파일·구글드라이브 문서를 모아 → '이상협 이사 관점'의 쟁점 브리핑과 발언안까지 자동 생성. 인터서브·누가회 이사회용 도구 각각 보유.
회의록
맥미니에 상시 가동되는 '베드로 줌봇'이 회의를 전사 → 요약 → 폴더에 자동 저장. 사람이 받아적지 않는다.
강의 1건
초안 텍스트 → 강의안 4종 → 슬라이드 → 팟캐스트·사전읽기 자료·피드백 설문까지 한 파이프라인. 준비가 며칠에서 몇 시간으로.
AI는 '비서 한 명'이 아니라 나를 복제·분업한 팀이다. 외주가 아니라 내가 직접 만들고, 만든 것을 공동체가 같이 쓴다.
Section 02

무엇을 구축했나 — 4개 층위

아래에서 위로 쌓인다. 정리(데이터)가 바닥, 공유(결과물)가 꼭대기.

  • Layer 1 — 기반
    세컨드 브레인 / 온톨로지 (Obsidian)
    노트 4,331개. 공식 온톨로지(엔티티 10종: 인물 119·성경 408·주제 127·책 103·강의 82)로 지식이 그래프가 됨. "단일 진실원(single source of truth)".
  • Layer 2 — 에이전트
    명명된 에이전트 + 스킬 (Claude Code)
    사역별 AI 페르소나 5종 + 재사용 스킬 13종. 한 번 만든 업무는 매번 다시 만들지 않고 호출만.
  • Layer 3 — 자동화
    자동화 인프라
    베드로 줌봇·n8n+Tailscale·카카오 알림·유튜브 업로드. 사람이 안 깨어 있어도 도는 부분.
  • Layer 4 — 공유
    만들어 나누는 결과물
    실사용 웹앱 + 강의·자료 HTML 웹 공개 19건 + GitHub 지속 공유. 내 노트와 도구가 곧 공동체 공개 자산.

실사용 웹앱 — 대화로 만든 도구가 실제 사역에서 돈다

자원봉사 모집·관리 웹앱. youngdoctor.kr/icmda-vol로 운영 중.
ICMDA 홍보 전략 자료

CMF 회원관리 앱
회원 데이터·소통 관리 앱, 현재 구성·진행 중.
핵심
코딩 전공이 아니어도, 사역을 아는 내가 직접 AI와 만들어 현장에 올렸다는 점이 핵심.

공개 위키 아카이브 — wiki.youngdoctor.kr/marktl : 세컨드 브레인(노트 4,331개)을 실제로 공개한 곳. 강의·연구·사역 노트가 누구나 볼 수 있는 공동체 자산으로 나간다.

Section 03

서브에이전트 — 'AI 팀'으로 일한다

핵심 차별점. 하나의 챗봇이 아니라, 총괄(오케스트레이터)이 역할별 전문가에게 위임하는 구조다.

🪨 베드로 (Peter) — 오케스트레이터 전체를 지휘하고, 일을 분해해 각 에이전트에 위임 ┌──────────┬──────────┬──────────┬──────────┐ 바울 누가 스데반 베살렐 인터서브 선교 클리닉·의료 교회 사역 소식지·창작

왜 '팀'이 비서보다 강한가

역할 분리 = 맥락 보존
각 에이전트는 자기 규칙(paul_rule.md, luke_rule.md 등)과 스킬을 가져, 사역별 말투·우선순위·보안 경계를 지킨다.
복잡한 일은 분해·병렬
3개 이상 파일·다단계 작업은 멀티 에이전트가 나눠 동시 처리.
검증 루프
결과를 다른 에이전트가 교차 검증, 자기개선 세션으로 도구가 스스로 나아진다.
"거듭난 베드로" 모드
평소엔 친절한 비서지만, 호출하면 독한 전략가로 변신: ① 소크라테스식 반론(기획의 맹점 3가지) ② 사전 사후분석(실패 시나리오) ③ 제1원리 검증. AI를 '예스맨'이 아니라 '반대 심문관'으로 쓰는 장치.
Section 04

핵심 포인트 6가지

교육 콘텐츠의 씨앗이 되는 원칙들.

1
AI보다 정리가 먼저
입력(온톨로지)이 구조화돼야 출력이 좋다.
2
외주가 아닌 내재화
사역을 아는 내부자가 만들어야 맥락이 산다.
3
반복은 스킬로 자산화
1회성 대화 → 재사용 가능한 도구.
4
보안은 설계로 푼다
인증정보는 저장소 밖, 민감 데이터는 로컬에.
5
작게·싸게 시작
큰 인프라가 아니라 구독료+학습시간이 진짜 비용.
6
만든 것은 공개·공유
GitHub/웹으로 공동체 자산화 → 신뢰·협업·교육의 토대.
Section 05

인터서브 공동체 R&D 제안

외부 도움엔 한계가 있다 — 선교 현장의 보안·신뢰·맥락은 외부 벤더가 채울 수 없다. 본질은 내부자 역량 구축이다.

3단계 모델

Phase 1
스터디 모임
월 1회, 펠로우 대상. 사례 공유 + 손으로 만드는 실습.
Phase 2
펠로우 내 전문가 컨설팅
역량 있는 펠로우가 1:1 / 소그룹 코칭. 외부가 아닌 내부 멘토.
Phase 3
공용 스킬·도구 라이브러리
GitHub로 운영. 이사회·회의록·강의 같은 공통 업무부터 오픈소스 자산화. 누구나 가져다 쓰고 개선.
첫 적용 지점
교육·교수 사역 / 학교시스템 행정 — 이미 2케이스 경험으로 효과가 가장 빨리 나는 영역.
도구를 만드는 건 싸다. 그러나 이 모델은 한 명이 아니라 여러 명의 펠로우를 키워야 작동한다. 강사·멘토의 시간, 실습 환경, 구독료, 모임 운영비는 실제 비용이다. "사람 투자 = 무료"가 아니라 사람을 키우는 데 지속적 재정이 든다는 점을 분명히 하고, R&D 예산을 사람·교육 라인에 배정해야 한다.
Section 06

"AI 시대, 인프라 예산" 질문에 답함

"AI 시대가 왔는데, 인프라 구축에 000원 책정 — 과연 괜찮을까?"
— 정OO 파트너, 총회 중 질문
구분 핵심
잘못된 프레임 AI를 큰 장비·인프라(capex) 프로젝트로 보는 것 → 돈은 쓰고 역량은 안 남는 함정.
비용의 이동 돈이 덜 드는 게 아니라 옮겨간다. 하드웨어(capex)에서 사람·교육·운영(opex)으로. 줄어드는 비용이 아니라 성격이 바뀌는 비용이다.
그래서 재정 사람을 여러 명 키워야 하기에 오히려 재정이 더 중요하다. 강사·멘토 사례비, 실습·구독, 모임 운영 — 이건 실비. R&D 예산은 '사람 키우는 예산'으로 잡아야 한다.
그 예산은? 장비 시드로는 충분. 다만 그 옆에 '사람·교육' 예산 라인이 반드시 같이 서야 한다. 장비만 사고 사람을 안 키우면 역량은 안 남는다.
미래조직 작은 R&D 펠로우십(사람+학습+소액 운영비)을 지속 재정으로 받친다. GitHub 공용 자산화로 중복 투자 없이 한 번의 개발을 공동체 전체가 공유 → 재정을 '도구'가 아니라 '사람'에 집중시키는 구조.
Section 07

교육·훈련 & 멘토링 파일럿

이번 주 파트너(선교사) 2명 멘토링 = 이 R&D 모델의 첫 파일럿. 반복 가능한 멘토링 템플릿으로 정형화한다.

  1. 시연
    내 사례 1~2개 라이브(이사회 준비 또는 줌봇).
  2. 진단
    각자 "가장 반복되는 업무 1개" 찾기.
  3. 공동제작
    그 업무를 작게 자동화 1건 함께 만들기.
  4. 공유
    결과물을 공용 GitHub/볼트에 올려 다음 펠로우가 재사용.
  5. 후속
    월 1회 스터디 모임으로 연결 (멘토링 → 정기 R&D).
